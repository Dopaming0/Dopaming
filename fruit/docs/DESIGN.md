# 🍑 과일사장님 — 작업 설계서

> 과일 위탁판매를 누구나 쉽고 편하게. 반복 업무 5가지를 하나씩 자동화한다.
> 작성일 2026-07-10 · 문서 버전 v1

---

## 1. 서비스 개요

### 1.1 문제

쿠팡 과일 위탁판매 셀러의 하루는 반복 작업의 연속이다.

1. 쿠팡윙에서 발주서(주문 목록)를 내려받아
2. 도매처별로 주문을 손으로 나눠 발주서를 만들어 보내고
3. 도매처가 보내온 송장 파일을 쿠팡 송장 업로드 양식에 옮겨 적고
4. 고객 문의(배송·품질·교환환불)에 비슷한 답변을 반복하고
5. 어떤 옵션이 팔리는지 감으로 판단한다

각 단계가 엑셀 복붙 노동이라 실수가 잦고, 초보 셀러의 진입 장벽이 된다.

### 1.2 타겟 사용자

- 쿠팡 위탁판매를 시작했거나 시작하려는 **1인 셀러**
- 엑셀에 익숙하지 않은 사람도 쓸 수 있어야 함 (전문용어 금지)
- 주 사용 환경: PC 브라우저(엑셀 업로드 작업), 모바일에서도 조회 가능해야 함

### 1.3 핵심 가치

| 가치 | 구현 방식 |
|---|---|
| 쉽다 | 토스·당근 스타일 — 한 화면 한 가지 일, 3단계 이내, 친근한 말투 |
| 안전하다 | 고객 개인정보(이름·전화·주소)가 담긴 파일을 **브라우저 밖으로 내보내지 않음** |
| 가볍다 | 가입·설치 없음. 링크 열면 바로 사용 |

## 2. 설계 원칙

### 2.1 UX 원칙 (토스·당근에서 배운 것)

1. **한 화면 한 가지 일** — 페이지마다 목적 하나. 홈은 허브 역할만
2. **단계는 3개 이내** — 올리기 → 확인/지정 → 받기
3. **먼저 해주고, 필요할 때만 묻는다** — 열 매핑은 자동 인식이 기본, 수정 UI는 접어둠
4. **기억해서 다음엔 안 묻는다** — 도매처 지정, 양식 설정은 localStorage에 저장
5. **전문용어 대신 사장님의 말** — "파싱 실패" ✗ → "주문 목록을 찾지 못했어요" ✓
6. **빈손으로 와도 체험 가능** — 모든 기능에 "샘플로 체험하기" 제공
7. **실수 방지** — 미지정·매칭 실패 건은 경고 배너로 개수까지 명시

### 2.2 기술 원칙

- **서버리스 우선**: 정적 파일 + 클라이언트 사이드 처리. GitHub Pages로 배포 (`/fruit/`)
- **빌드 도구 없음**: 순수 HTML/CSS/JS. 누구나 파일 열어 수정 가능
- **라이브러리는 로컬 번들**: SheetJS 등 CDN 의존 없이 저장소에 포함 (오프라인 동작, 정책 차단 회피)
- **개인정보 원칙**: 주문·고객 데이터는 서버 전송 금지. 집계 결과 등 비식별 데이터만 localStorage 저장 허용
- **AI가 필요한 기능(③④)은 후순위 단계에서 API 연동 여부 결정** — 그 전까지는 규칙 기반으로 가치 제공

### 2.3 공통 모듈 (기능이 늘 때 재사용)

| 모듈 | 현재 위치 | 역할 |
|---|---|---|
| 디자인 시스템 | `assets/style.css` | 색·타이포·카드·버튼·업로드 영역·알림 배너 |
| 엑셀 리더 | `assets/order.js` → 공용화 예정 | 파일 드롭존, 헤더 행 자동 탐색(앞 15행 스코어링), 키워드 규칙 기반 열 매핑, 수동 매핑 UI |
| 저장소 | localStorage | 네임스페이스 `gwail.*` (아래 6장 스키마) |

> ②번 기능 착수 시 `order.js`의 드롭존·헤더 탐색·매핑 로직을 `assets/excel-common.js`로 분리한다.

## 4. 기능별 상세 설계

### ① 발주서 변환 — ✅ 완료 (PR #4)

**쿠팡 발주서 엑셀 → 도매처별 발주서 엑셀**

- 3단계: 올리기 → 옵션마다 도매처 지정 → 도매처별/통합 다운로드
- 헤더 자동 인식 8열: 수취인이름·전화·우편번호·주소·옵션명·수량·배송메세지·주문번호
- 옵션→도매처 매핑은 localStorage에 기억, 다음 파일부터 자동 채움
- 출력 양식: `받는분 성명 | 전화번호 | 우편번호 | 주소 | 상품명(옵션) | 수량 | 배송메세지 | 쿠팡 주문번호`

**개선 백로그** (사용 피드백 후 착수)
- [ ] 도매처별 커스텀 출력 양식 (열 순서·명칭을 도매처 요구에 맞게 저장)
- [ ] 묶음배송번호 기준 동일 수취인 병합 옵션
- [ ] 스마트스토어 발주서 지원 (헤더 규칙 추가만으로 가능)

### ② 송장 자동 입력 — 🔜 다음 착수

**도매처 송장 파일 → 쿠팡 송장 업로드 양식**

①의 정확한 역방향. 도매처가 보내온 "발주서 + 운송장번호" 파일을 읽어, 쿠팡윙에 올릴 송장 엑셀을 만들어 준다.

**입력 2개**
1. 쿠팡 발주서 (①에서 쓴 것과 동일 — 주문번호·수취인 기준 데이터)
2. 도매처 송장 파일 (양식 제각각 — 최소한 수취인 이름/전화/주소 + 운송장번호 + 택배사)

**매칭 전략** (우선순위 순)
1. **쿠팡 주문번호**가 도매처 파일에 살아있으면 그걸로 매칭 (①의 출력에 주문번호를 넣어둔 이유)
2. 없으면 **수취인 이름 + 전화번호 뒤 4자리** 복합키
3. 그래도 못 찾으면 **이름 + 주소 앞부분** 유사 매칭 → "확인 필요" 목록으로 분리

**출력**: 쿠팡 송장 업로드 양식 (`주문번호 | 택배사 | 운송장번호`). 택배사명은 쿠팡 인정 명칭으로 정규화(예: "CJ", "cj대한통운" → "CJGLS" 코드표).

**엣지 케이스**
- 동명이인 + 같은 전화번호 (가족 주문): 주소까지 비교, 그래도 겹치면 확인 필요 목록
- 부분 발송(주문 10건 중 송장 7건): 미발송 3건을 명시적으로 보여줌
- 한 주문 분리배송(송장 2개): 첫 송장 채택 + 경고 표시

**완료 기준(DoD)**: 샘플 도매처 송장 3가지 양식(열 순서 다름)에 대해 매칭률 100%, 실패 건 0이 아닌 경우 확인 필요 목록 노출. Playwright 자동 검증.

> ⚠️ 실제 도매처 송장 샘플을 받으면 매칭 규칙을 실데이터로 보정한다. **사용자에게 샘플 요청 중.**

### ③ 썸네일 · 상세페이지 제작

**사진 몇 장 + 문구 입력 → 판매용 이미지**

- 1차 (규칙 기반, 서버리스): Canvas API 템플릿 엔진
  - 썸네일: 1000×1000px. 템플릿 4종(깔끔/시즌/할인강조/선물세트) × 사진 1장 + 상품명·무게·가격 텍스트
  - 상세페이지: 세로 스크롤 이미지(가로 860px). 블록 조립식 — 표지/산지 소개/실측 사진/보관법/배송 안내 블록을 순서대로 쌓아 1장의 긴 PNG로 내보내기
  - 과일별 프리셋(참외·수박·멜론·복숭아…): 색상·문구 예시 자동 추천
- 2차 (선택): AI 카피라이팅·배경 제거 API 연동 — 비용·키 관리 필요하므로 별도 결정

**DoD**: 사진 2장과 상품명만으로 3분 안에 썸네일 + 상세페이지 PNG 다운로드 가능.

### ④ CS 자동응답

- 1차 (규칙 기반): **답변 생성기** — 문의 유형 선택(배송 지연/파손·무름/맛 불만/교환환불/영수증) → 상황 입력(주문일, 상품명) → 정중한 답변 문구 생성 → 복사 버튼. 내 가게 정책(환불 기준, 발송 요일)을 localStorage에 저장해 답변에 자동 반영
- 2차 (AI): 문의 원문을 붙여넣으면 유형 분류 + 맞춤 답변 생성. Claude API 연동 — **개인정보(고객명·주소) 마스킹 후 전송** 원칙, API 키는 사용자 본인 키 입력 방식(서버 없이)
- 3차 (장기): 쿠팡윙 연동 자동 응답은 공식 API 제약 조사 후 판단

**DoD(1차)**: 5개 유형 × 3개 톤(정중/친근/간결) 답변 생성, 가게 정책 반영 확인.

### ⑤ 오늘의 판매 분석

**쿠팡 판매/주문 엑셀 → 옵션별 현황 대시보드**

- 입력: 쿠팡윙 주문 목록(①과 같은 파일 재사용 가능) 또는 매출 내역 엑셀
- 집계: 옵션별 판매량·매출·주문 건수, 시간대별 주문 분포
- 추이: 업로드할 때마다 일별 스냅샷을 localStorage에 저장(개인정보 제외, 집계값만) → 전일 대비 증감, 7일 추이 미니 차트
- 표현: 카드형 요약(오늘 총 주문/매출) + 옵션별 순위 리스트 + 간단한 바/라인 차트 (외부 차트 라이브러리 없이 SVG 직접 그리기 또는 경량 라이브러리 로컬 번들)

**DoD**: 발주서 업로드만으로 "오늘 뭐가 잘 팔렸나"를 10초 안에 파악 가능. 이틀치 업로드 시 증감 표시.

## 6. 데이터 설계 (localStorage)

네임스페이스 `gwail.*`. 모든 값은 JSON.

| 키 | 내용 | 사용 기능 |
|---|---|---|
| `gwail.vendorMap` | `{ 옵션명: 도매처명 }` | ① (사용 중) |
| `gwail.courierAlias` | `{ 입력명: 쿠팡 택배사 코드 }` 사용자 보정값 | ② |
| `gwail.shopPolicy` | 가게 정책(발송 요일, 환불 기준, 인사말) | ④ |
| `gwail.dailyStats` | `[{ date, options: [{name, qty, revenue}] }]` 집계값만, 개인정보 없음 | ⑤ |
| `gwail.templates` | 상세페이지 블록 구성·색상 프리셋 | ③ |

- 고객 식별 정보(이름·전화·주소·주문번호)는 **어떤 키에도 저장하지 않는다**
- 용량 한계(~5MB) 대비: `dailyStats`는 90일 초과분 자동 삭제

## 8. 검증 방식

- 기능마다 **샘플 데이터 생성기**를 내장 (체험 겸 테스트 픽스처)
- Playwright 헤드리스 스모크 테스트: 업로드 → 처리 → 다운로드 파일을 다시 파싱해 내용 검증
- 머지 전 체크: 모바일 뷰포트(420px) 스크린샷 확인, JS 콘솔 에러 0건
