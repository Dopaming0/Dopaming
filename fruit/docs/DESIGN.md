## 1. 서비스 개요

**문제** — 쿠팡 과일 위탁 셀러의 하루는 엑셀 복붙 노동이다. ① 발주서를 도매처별로 쪼개고 ② 도매처 송장을 쿠팡 양식으로 옮기고 ③ 판매 잘 되는지 감으로 보고 ④ 비슷한 CS에 매번 답한다. 실수가 잦고 초보의 진입장벽이 된다.

**타겟** — 쿠팡 위탁판매 1인 셀러. 엑셀에 익숙하지 않아도 됨(전문용어 금지). PC 브라우저로 엑셀 작업, 모바일로 조회.

**핵심 가치**

| 가치 | 구현 |
|---|---|
| 쉽다 | 한 화면 한 가지 일 · 3단계 이내 · 돈나올구멍이 말투 |
| 안전하다 | 고객 개인정보(이름·전화·주소)를 **브라우저 밖으로 안 내보냄** |
| 가볍다 | 가입·설치 없음. 링크 열면 바로 |

## 2. 브랜드 & 톤앤매너 — 돈나올구멍이 × 토스

### 2.1 캐릭터: 돈나올구멍이

레퍼런스(`돈나올구멍이.png`) 기준 정의.

- **생김새**: 손그림/크레용 질감의 둥근 **먹색(숯빛) 몸체**, 큰 흰 눈 + **코발트블루 눈동자**, 작은 미소, 가느다란 **파란 막대 팔다리**.
- **의미**: '돈 나올 구멍'을 귀엽게 의인화. 어둡고 둥근 '구멍'이지만 웃고 있어 위협적이지 않다.
- **성격/무드**: 순박하고 엉뚱하지만 **사장님 편에 서서 같이 돈 벌자는 든든한 파트너**. 완벽하지 않은 손그림 텍스처의 따뜻함이 "누구나 쉽게"라는 메시지와 맞는다.
- **역할**: 마스코트 겸 안내자. 스플래시·로딩·성공·빈손 화면에 등장해 상태를 대신 말한다.
- **자산**: `assets/brand/donnaol.svg` (라이트/다크 겸용). 손그림 느낌을 살리려 SVG는 러프한 획·텍스처를 살린 형태로 제작.

### 2.2 토스에서 가져올 것 / 안 가져올 것

**가져온다(패턴·컨벤션)**
- 다크모드 기반 **몰입형 딜라이트 순간**과 글로시 3D 오브젝트로 상태 표현
- **일렉트릭 블루** 단일 강조색(돈나올구멍이 팔다리색과 통일)
- 굵은 **2줄 헤드라인** + 하단 **풀폭 파란 알약 CTA**(화면당 주 액션 1개)
- 카운트다운·토스트 등 상태 UI, 즐거운 로딩/성공 연출
- **이름 부르고 돈으로 후킹하는 따뜻한 반말체 카피** ("돈 뽑는 중..", "혜택을 드릴게요")

**안 가져온다**
- 토스 로고·고유 3D 일러스트를 **그대로 복제 금지**(IP). 같은 '정신'으로 우리 자산을 새로 만든다.
- **전면 다크모드 강제 금지** — 엑셀 작업 화면은 명료성을 위해 라이트 기본(§2.3).
- 금융앱 특유의 절제된 중립 톤 대신 **더 능청스럽게**.

### 2.3 비주얼 시스템 — 핵심 결정: 라이트 작업 / 다크 딜라이트

두 레퍼런스(손그림 캐릭터 = 라이트, 토스 연출 = 다크)를 이렇게 화해시킨다.

| 화면 종류 | 모드 | 이유 |
|---|---|---|
| **작업 화면** (파일 올리기·도매처 지정·매칭 확인·분석 표) | **라이트 기본** (흰 배경, 큰 숫자, 높은 대비) | 엑셀 노동을 오래 봐도 안 피로. 데이터 가독성 우선 |
| **딜라이트 순간** (스플래시·처리 로딩·성공 축하·빈손 안내) | **다크 글로시** (검은 배경 + 발광하는 돈나올구멍이/₩·과일 3D) | 토스식 "돈 뽑는 중.." 연출을 발주 변환/송장 매칭 로딩에 그대로 |

- 캐릭터 배치: 라이트에선 그대로, 다크에선 **은은한 외곽광(글로우)** 또는 밝은 발판 위에.
- 즉 앱의 뼈대는 밝고 명료하되, "일이 처리되는 순간"만 토스처럼 화려하게 → 초보도 안 헤매고, 성취감은 준다.

### 2.4 디자인 토큰 (`assets/style.css`)

값은 캐릭터 팔다리색에서 추출·확정. 아래는 초안.

```css
:root{
  /* Brand */
  --blue:        #2E2FF2;   /* 돈나올구멍이 팔다리 = 브랜드/주 액션/양(+) 수치 */
  --blue-press:  #2323C4;
  --ink:         #141414;   /* 본문·캐릭터 몸체 */
  --peach:       #FF8A5B;   /* 과일 액센트 — 일러스트에만, UI 크롬엔 X */

  /* Neutral (라이트) */
  --bg:      #FFFFFF;  --surface: #F6F7F9;  --line: #ECEEF1;
  --text:    #141414;  --text-sub:#6B7280;

  /* Dark delight */
  --bg-dark: #0B0B0F;  --dark-sub:#9AA0AA;

  /* Semantic */
  --up:   #2E2FF2;     /* 증가/좋음 = 브랜드 블루 */
  --down: #E5484D;     /* 감소/경고 */

  /* Shape & motion */
  --r-card: 20px;  --r-btn: 16px;  --r-chip: 999px;
  --ease: cubic-bezier(.2,.8,.2,1);  --dur: 200ms;
}
```

- **타이포**: Pretendard(한글 클린 산세리프, 로컬 번들). 헤드라인 800, 본문 500. **금액·개수는 `font-variant-numeric: tabular-nums`**.
- **모션**: 150–250ms `--ease`. 딜라이트 순간만 스프링/발광 애니. 그 외는 미세 인터랙션(눌림·페이드).

### 2.5 보이스 & 마이크로카피 규칙

**Do** — 사장님 호칭 · 돈/장사 관점 한 줄 · 짧고 자신감 · 가끔 능청.
**Don't** — 이모지 도배 · 사행성/과장("무조건 대박") · 전문용어 · 고객 비하.

| 상태 | ✗ (금지) | ✓ (돈나올구멍이 보이스) |
|---|---|---|
| 스플래시 | 시스템 초기화 중 | **돈 나올 구멍, 열어드릴게요** |
| 처리 로딩 | 파일 파싱 중… | **도매처별로 착착 나누는 중..** |
| 성공 | 변환 완료 | **사장님, 발주서 4개 다 만들었어요. 바로 보내면 돼요** |
| 경고(미지정) | 매핑되지 않은 항목 존재 | **잠깐! 도매처 아직 안 정한 옵션이 3개 있어요** |
| 빈손 | 데이터 없음 | **처음이세요? 샘플로 5초 만에 구경해봐요** |
| 에러 | 헤더 인식 실패 | **엇, 주문 목록을 못 찾았어요. 쿠팡윙에서 받은 그 파일 맞을까요?** |

## 3. 아키텍처 원칙 (기술)

- **서버리스 우선**: 정적 파일 + 클라이언트 처리. GitHub Pages 배포(`/fruit/`). (완전 자동화 단계에서만 백엔드 도입 — §7)
- **빌드 도구 없음**: 순수 HTML/CSS/JS(ES Modules). 누구나 파일 열어 수정.
- **라이브러리 로컬 번들**: SheetJS 등 CDN 의존 없이 저장소 포함(오프라인·정책 차단 회피).
- **개인정보 경계**: 주문·고객 데이터 서버 전송 금지. 비식별 집계값만 localStorage 허용. → 이 불변식은 §10에서 **자동 테스트로 강제**.
- **규칙기반 먼저**: AI가 필요한 기능(이미지 카피·CS 분류)도 1차는 규칙기반으로 가치 제공, API는 뒤에서 결정.

### 3.1 순수 코어 + IO 셸 (품질의 토대)

돈이 걸린 로직(파싱·라우팅·매칭·정규화·집계)을 **DOM/IO 없는 순수 함수**로 `assets/core/`에 격리한다. 파일 읽기·드롭존·localStorage·다운로드는 `assets/io/`의 얇은 어댑터가 맡는다. 이 분리 하나로 (1) 테스트 가능 (2) 기능 간 재사용 (3) 버그 격리가 동시에 해결된다. (효과는 §10-P0)

### 3.2 공통 모듈

| 모듈 | 위치 | 역할 |
|---|---|---|
| 디자인 시스템 | `assets/style.css` | 토큰·카드·버튼·업로드·배너·토스트 |
| 엑셀 코어 | `assets/core/excel.js` | 헤더행 자동 탐색(앞 15행 스코어링), 키워드 규칙 열 매핑 |
| 저장소 | `assets/io/storage.js` | `gwail.*` 네임스페이스 + **개인정보 차단 지점**(§8) |

> A-1(`order.js`)의 드롭존·헤더 탐색·매핑 로직을 A-2 착수 시 `core/excel.js` + `io/dropzone.js`로 분리한다.

## 5. 기능 그룹별 설계

각 기능: **목적 · 입력 · 처리(규칙기반) · 출력 · 엣지 · DoD · 데이터/검증**. UI 세부는 구현 시 확정.

### A. 주문처리

발주서·송장을 하나의 흐름으로 묶는다. **A-1 출력이 A-2 입력의 매칭 키를 제공**(그래서 A-1 출력에 쿠팡 주문번호를 넣는다). A·C는 같은 WING 발주서 파일로 이어지는 하나의 일일 워크플로.

#### A-1. 발주서 변환 — ✅ 완료 (PR #4)

- **입력** 쿠팡 발주서 엑셀 → **출력** 도매처별 발주서 엑셀
- **3단계**: 올리기 → 옵션마다 도매처 지정 → 도매처별/통합 다운로드
- **자동 인식 8열**: 수취인이름·전화·우편번호·주소·옵션명·수량·배송메세지·주문번호
- 옵션→도매처 매핑은 `gwail.vendorMap`에 기억, 다음 파일부터 자동 채움
- **출력 양식**: `받는분 성명 | 전화번호 | 우편번호 | 주소 | 상품명(옵션) | 수량 | 배송메세지 | 쿠팡 주문번호`
- **백로그**(피드백 후): 도매처별 커스텀 출력 양식 · 묶음배송번호 병합 · 스마트스토어 발주서 지원(헤더 규칙 추가)

#### A-2. 송장 자동 입력 — 🔜 다음 착수

A-1의 역방향. 도매처가 보낸 "발주서+운송장번호"를 읽어 쿠팡윙 송장 업로드 엑셀을 만든다.

- **입력 2개**: ① 쿠팡 발주서(A-1과 동일) ② 도매처 송장 파일(양식 제각각: 최소 이름/전화/주소 + 운송장번호 + 택배사)
- **매칭 전략**(우선순위) — `core/invoice-match.js`
  1. **쿠팡 주문번호**가 도매처 파일에 살아있으면 그걸로
  2. 없으면 **이름 + 전화 뒤 4자리** 복합키
  3. 그래도 없으면 **이름 + 주소 앞부분** 유사 매칭 → **"확인 필요" 목록**으로 분리
- **출력**: `주문번호 | 택배사 | 운송장번호`. 택배사명은 쿠팡 인정 코드로 정규화(`core/courier.js`, 예: "CJ"·"cj대한통운" → `CJGLS`)
- **엣지**: 동명이인+같은 전화(가족) → 주소까지 비교 후 확인 필요 / 부분발송(10건 중 7건) → 미발송 3건 명시 / 한 주문 분리배송(송장 2개) → 첫 송장 채택 + 경고
- **DoD**: 도매처 송장 3양식(열 순서 다름) 매칭률 100%, 실패 시 확인 필요 목록 노출. Playwright 검증.
- ⚠️ **실데이터 필요**: 실제 도매처 송장 샘플 확보 후 매칭 규칙 보정. (사용자에게 요청 중)

### B. 이미지 생성기 (썸네일 · 상세페이지)

사진 몇 장 + 문구 → 판매용 이미지. 100% 클라이언트(Canvas). 개인정보 무관 → 마스코트 딜라이트 적극 활용.

- **B-1 썸네일**: 1000×1000px. 템플릿 4종(깔끔/시즌/할인강조/선물세트) × 사진 1장 + 상품명·무게·가격 텍스트
- **B-2 상세페이지**: 세로 스크롤 이미지(가로 860px). **블록 조립식** — 표지/산지 소개/실측 사진/보관법/배송 안내 블록을 쌓아 1장의 긴 PNG로 내보내기
- **과일 프리셋**(참외·수박·멜론·복숭아…): 색상·문구 예시 자동 추천 → `gwail.templates`
- **처리**: `core/`엔 레이아웃 계산(블록→좌표/크기)만 순수 함수로, Canvas 렌더는 `ui/image.js`
- **2차(선택)**: AI 카피라이팅·배경 제거 API(비용·키 관리 별도 결정 §7)
- **DoD**: 사진 2장 + 상품명만으로 **3분 내** 썸네일 + 상세페이지 PNG 다운로드

### C. 오늘 판매 분석기

쿠팡 판매/주문 엑셀 → 옵션별 대시보드. **A와 같은 발주서 파일 재사용** → 저비용·고가치.

- **입력**: 쿠팡윙 주문 목록(A와 동일 파일) 또는 매출 내역 엑셀
- **집계**(`core/stats.js`): 옵션별 판매량·매출·주문 건수, 시간대별 분포
- **추이**: 업로드마다 일별 스냅샷을 `gwail.dailyStats`에 저장(**집계값만, 개인정보 X**) → 전일 대비 증감, 7일 미니 차트
- **표현**: 카드 요약(오늘 총 주문/매출) + 옵션 순위 리스트 + 바/라인 차트(외부 라이브러리 없이 **SVG 직접** 또는 경량 로컬 번들)
- **DoD**: 발주서 업로드만으로 "오늘 뭐가 잘 팔렸나"를 **10초 내** 파악. 이틀치 업로드 시 증감 표시.

### D. CS 응답기

- **1차(규칙기반) 답변 생성기**: 문의 유형 선택(배송 지연/파손·무름/맛 불만/교환환불/영수증) → 상황 입력(주문일·상품명) → 정중한 답변 문구 생성 → 복사. 가게 정책(환불 기준·발송 요일·인사말)을 `gwail.shopPolicy`에 저장해 자동 반영
- **2차(AI)**: 문의 원문 붙여넣기 → 유형 분류 + 맞춤 답변. **개인정보(고객명·주소) 마스킹 후 전송** 원칙, API 키는 사용자 본인 키 입력(서버 없이)
- **3차(장기)**: 쿠팡윙 연동 자동 응답 — 공식 문의조회 API 제약 조사 후 판단(§7)
- **DoD(1차)**: 5개 유형 × 3개 톤(정중/친근/간결) 생성, 가게 정책 반영 확인

## 6. 단계별 로드맵 (Phase)

**원칙: 한 Phase = 한 PR = 한 기능. 각 PR은 CI(lint→typecheck→test→e2e) 통과 후 머지.**
각 Phase는 아래 게이트를 통과해야 다음으로 간다.

| Phase | 목표 | 산출물 | 착수 조건 | 완료 게이트(DoD) | 상태 |
|---|---|---|---|---|---|
| **P0** | 서비스 셸 + 디자인 시스템(§2.4) + 홈 + **품질 뼈대**(core/io 분리·jsconfig·eslint·CI) | `index.html` `style.css` `.github/ci.yml` | — | 홈 렌더 · CI 그린 · 토큰 적용 | ✅ 셸/홈 완료, 품질 뼈대 이번에 추가 |
| **P1** | **A 주문처리 완성** — 공용 엑셀 코어 분리 + A-2 송장 자동 입력 | `core/excel.js` `core/invoice-match.js` `core/courier.js` `invoice.html` | 도매처 송장 실샘플 확보 | 3양식 매칭률 100% · 확인 필요 목록 · e2e 통과 | 🔜 다음 |
| **P2** | **C 판매 분석기** (같은 입력 재사용, 저비용·고가치) | `core/stats.js` `stats.html` | P1 엑셀 코어 재사용 | 10초 내 요약 · 이틀치 증감 | |
| **P3** | **B 이미지 생성기** 1차(템플릿) | `image.html` | 프리셋·템플릿 확정 | 3분 내 PNG 2종 | |
| **P4** | **D CS 응답기** 1차(답변 생성기) | `cs.html` | 가게 정책 스키마 확정 | 5유형×3톤 · 정책 반영 | |
| **P5** | **API 완전 자동화 검토·도입** | 백엔드 프록시 등(§7) | **백엔드 도입 결정 게이트 통과** | 자동화 지점별 검증 | |

> **왜 C(P2)를 B(P3)보다 앞에?** A·C는 같은 발주서 파일로 이어지는 하나의 일일 워크플로라, 묶어야 체감 가치가 크다. B는 독립 작업이라 뒤로 미뤄도 무방.
> **왜 4그룹인데 Phase는 A를 하나로?** A-1은 이미 완료. A-2만 남았고 A-1과 데이터·코어를 공유하므로 한 Phase(P1)로 처리.

## 7. API 완전 자동화 로드맵 (미래)

> 현재는 **브라우저 단독 + 수동 업로드/다운로드**라 서버·개인정보 유출이 0이다. 완전 자동화는 이 원칙을 깨는 결정이므로 **P5의 게이트**로 다룬다.

### 7.1 쿠팡 WING OpenAPI로 가능한 것 (확인됨)

`developers.coupangcorp.com` 기준. WING에서 직접 키 발급(판매자정보 › 추가판매정보 › OPEN API 키), 별도 제휴 절차 없음.

| 기능 | 관련 API | 자동화되는 우리 기능 |
|---|---|---|
| 발주서 조회 | 발주서 목록 조회 | A-1 입력 자동 수집, C 집계 자동화 |
| 송장 업로드 | 송장업로드 처리 | A-2 출력 자동 반영 |
| CS 문의 조회 | 쿠팡고객센터 문의조회 | D 문의 자동 수집 |
| 반품/취소 | 반품·취소 요청 조회 | D 응대 범위 확장 |

**제약(설계에 직결)**
- **HMAC 서명 + 업체코드/AccessKey/SecretKey** 필요 → SecretKey를 **브라우저에 못 실음**(서명 노출).
- **IP 허용목록 필수** — 미등록 IP는 `403 FORBIDDEN Not allowed IP` → **고정 IP 서버 필요**.
- **속도 제한** 있음. 송장 업로드는 **상품준비중 상태 주문만** 가능, 6개월 내 중복 송장번호 에러 주의.

### 7.2 핵심 결정 게이트: 백엔드 도입

위 두 제약(SecretKey 은닉 + 고정 IP)은 곧 **"서버리스·무서버" 원칙을 깬다**는 뜻이다. 완전 자동화하려면 백엔드 1개가 불가피하다. 선택지:

| 옵션 | 개인정보 | 비용/운영 | 비고 |
|---|---|---|---|
| (a) 초경량 자체 서버(고정 IP) 프록시 | 서버 경유 | 상시 서버 비용 | 가장 표준적 |
| (b) 서버리스 함수 + 고정 egress IP | 서버 경유 | 종량 | egress IP 고정 설정 필요 |
| (c) **사용자 PC 로컬 에이전트** | **로컬에만 머묾** | 사용자 실행 | 개인정보 원칙 최대 보존 · 별도 진행 중인 Claude Code 에이전트와 접점 |

> 개인정보 원칙을 지키려면 (c)가 이념적으로 가장 정합. 다만 사용자 실행 UX 부담이 있으니, 반자동은 브라우저 UI + (a/b) 프록시로, 완전 자동은 (c) 로컬 에이전트로 나누는 하이브리드도 가능.

### 7.3 단계적 전환

1. **반자동**: 브라우저 UI 유지 + 백엔드가 발주서 조회·송장 업로드 대행(사람이 버튼 누름)
2. **완전 자동**: 스케줄러가 `발주 폴링 → 라우팅 → 도매처 발송 → 송장 회수 → 쿠팡 업로드`
3. **불변 원칙**: **금전 영향 액션(발주 발송·송장 확정)은 사람 확인 게이트 유지.** 자동은 준비까지, 확정은 사람.

## 8. 데이터 설계 (localStorage)

네임스페이스 `gwail.*`. 모든 값 JSON.

| 키 | 내용 | 기능 |
|---|---|---|
| `gwail.vendorMap` | `{ 옵션명: 도매처명 }` | A-1(사용 중) |

## 11. 검증 방식 (요약)

- 기능마다 **샘플 데이터 생성기** 내장(체험 겸 테스트 픽스처)
